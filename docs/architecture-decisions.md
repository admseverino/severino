# Architecture Decisions

Locked-in technology choices that shape the implementation. Anything not listed here is open and can be decided per milestone. If a decision is reversed, update this file in the same PR that reverses it.

---

## 4.1 ORM + Migrations: Drizzle

**Decision:** Use [Drizzle ORM](https://orm.drizzle.team/) with the `pg` driver for both schema definition and migrations. All of that lives in a **root workspace package** `packages/db` (same idea as [`reference-code/packages/db`](../reference-code/packages/db)), not under `hidrosync-service/lib/`. The Next app (`hidrosync-service`) depends on the package and imports the typed client, schema, and shared query helpers from there.

**Why:**

- **Typed queries without a runtime layer.** Drizzle generates TS types from the schema; queries are plain SQL-builder calls that compile to predictable SQL. No N+1 surprises, no proxy objects.
- **Plays well with the polymorphic `linked_meters` table.** Drizzle lets us write the discriminator-based join in raw-ish SQL (or via a view) without fighting an ORM that wants every relation to be a typed FK.
- **Migration tooling that fits a single-developer cadence.** `drizzle-kit generate` produces SQL files we commit; `drizzle-kit migrate` applies them. Cloud Build can run the migrate step (from `packages/db` or a root workspace script) before the Cloud Run deploy.
- **No heavy runtime.** Drizzle's runtime is small (~10 KB) and tree-shakable, which matters for Cloud Run cold-starts.
- **One DB boundary for the monorepo.** A dedicated package avoids duplicating schema or a second Drizzle entry point inside the app; future services or jobs can reuse the same package if they talk to the same database.

**Why not the alternatives:**

- **Prisma** — fights the polymorphic `linked_meters` shape (no clean way to model `target_kind ∈ {unit, group, condo}` without three nullable FKs or three relation tables). Heavier runtime. Migration generation is opinionated.
- **Raw SQL only** — fine for the MVP, but loses type safety on joins and forces a hand-rolled migration runner. Drizzle is barely more setup and gives both.
- **Knex / Kysely** — Kysely is a strong runner-up; Drizzle wins on schema-as-source-of-truth (the schema *is* the TS file) and migration generation.

**Conventions:**

- Schema files live in `packages/db/src/schema/<context>.ts` (one per bounded context: `condos.ts`, `meters.ts`, `readings.ts`, etc.). A barrel `packages/db/src/schema/index.ts` re-exports everything the app and helpers need.
- Cross-cutting **query helpers** (period guards, scope joins, anything reused across `hidrosync-service/modules/`) live in `packages/db/src/queries/` (or equivalent) so they stay typed against the same schema.
- Migrations live in `packages/db/migrations/`. Generated by `drizzle-kit generate --name <verb-noun>` with `drizzle.config.ts` at the package root. Always commit the SQL — never edit a migration after it's been merged to `main`.
- The `meter_scope_resolved` view (see [4.5](#45-polymorphic-linked_meters--resolved-via-a-sql-view)) is a hand-written migration in that same folder, not generated.
- Prefer `db.transaction(async (tx) => …)` for any operation touching more than one table; the audit-log writer takes the same `tx` so the audit row is rolled back if the mutation is. Import `db` from `packages/db` (or pass `tx` from a single call site); do not instantiate a second pool in the app.

## 4.2 Period State Machine: Hand-Rolled

**Decision:** Implement the period state machine as a small TS module — a transition table + a guard function. No state-machine library.

**Why:**

- The machine has six states (`scheduled → reading_open → review → closed → billed → archived`) and two backward transitions (reopen: `archived → review`). That's a 6×6 transition table with maybe 8 entries. xstate or similar would be more code than the machine itself.
- Each transition has business preconditions (e.g. "no unresolved readings" for `review → closed`) that are easier to read inline than as guard objects.
- Auditability — we want the transition function to be the **single chokepoint** that writes both the `periods.state` update and the `audit_log` row, in the same transaction. Direct control of that flow is simpler than configuring a library to do it.

**Conventions:**

- One file: `hidrosync-service/modules/periods/transitions.ts`.
- Public API: `transition(periodId, to: PeriodState, { actor, reason }): Promise<Period>`.
- Internal: a `TRANSITIONS` const map of `{ from, to, guard }` triples. Every transition runs `assertGuard()` then `writeAudit()` then `UPDATE periods SET state = ...` inside one `db.transaction` (Drizzle client from `packages/db`).
- Tests live in `hidrosync-service/modules/periods/__tests__/transitions.test.ts` and exercise every legal and illegal transition.

## 4.3 Reading Pipeline: QR Detection + AI Vision via OpenRouter

**Decision:** Use a QR/barcode library on the client for **identifying which meter** is being read, and call **OpenRouter** with a vision-capable model to **extract the numeric value** from the meter photo.

This is the single trickiest feature in the system, so it has its own document — see [`meter-reading-pipeline.md`](./meter-reading-pipeline.md) for the full design (pipeline diagram, prompt shape, adapter interface, failure modes, cost, privacy).

**Headline points:**

- **No traditional OCR** (Vision API / Textract / Tesseract) in the value-extraction path. AI vision models handle the analog-dial / multi-display / partially-occluded cases far better than OCR.
- **Adapter pattern** so the OpenRouter call is one swappable implementation. Mock for dev/tests, real for prod, easy to add a fallback model.
- **AI value and operator value are stored separately** on the reading row — the operator's confirmed value is the source of truth, the AI value is kept for accuracy auditing.
- **Confidence drives anomaly flagging** — already specced in `condo_config.ocr_confidence_floor` (we'll keep the column name even though it's not OCR anymore, to avoid a churny rename).

## 4.4 Cron: Cloud Scheduler → `/api/scheduler/tick`

**Decision:** Use **Google Cloud Scheduler** to call a single authenticated HTTP endpoint `/api/scheduler/tick` on the main app once per day. The endpoint advances every condo's period state machine based on its `condo_config`.

**Why:**

- **Already on GCP.** `Infrastructure-plan.md` commits the project to Cloud Run; Cloud Scheduler is the native cron primitive there. No new infra category to manage.
- **Logic stays in the app.** The endpoint imports `hidrosync-service/modules/periods/transitions.ts` directly. No SQL-level cron (`pg_cron`) means no duplicated state-machine logic in PL/pgSQL.
- **Easy to replay.** The endpoint accepts an optional `?date=YYYY-MM-DD` (sysadmin-only) so we can re-run a missed tick or fast-forward in a dev/staging condo.
- **Auth is straightforward.** Cloud Scheduler signs requests with an OIDC token tied to a dedicated service account; the endpoint verifies the token and rejects anything else.

**Why not the alternatives:**

- **`pg_cron`** — duplicates the state machine in PL/pgSQL, harder to test, and gives Postgres the responsibility for time which it shouldn't have.
- **In-app `setInterval`** — Cloud Run scales to zero. There is no long-lived process to host an interval.
- **Cloud Tasks** — overkill for a daily tick; useful later for per-period work queues if we need them.

**Conventions:**

- One Cloud Scheduler job: `hidrosync-scheduler-tick-dev`, daily at 03:00 in the condo's relevant time zone (start with `America/Sao_Paulo`).
- Endpoint is idempotent: hitting it twice on the same day is a no-op (each transition checks the current state first).
- Endpoint logs every transition it performs and emits one structured log line per condo, even when no transition fires (so we can see "nothing to do" days clearly).

## 4.5 Polymorphic `linked_meters` — Resolved via a SQL View

**Decision:** Keep `linked_meters` as the polymorphic source of truth (`meter_id`, `target_kind`, `target_id`), and write a SQL view `meter_scope_resolved` that flattens each row into the **set of `unit_id`s the meter actually covers**. All consumption math joins against the view, not the raw table.

**Why:**

- The polymorphic shape is exactly what the spec asks for — submeters point to units, masters point to groups / the condo / a custom unit set. A relational alternative (three nullable FKs, or three join tables) would be uglier and harder to extend.
- **Consumption math is the same regardless of master scope.** Every common-area calculation is `master_value − Σ(submeters in master's coverage)`. Encoding "coverage" in a view means consumption code is one query, not a `CASE` over `target_kind`.
- **Reusable everywhere.** Anomaly detection, billing CSV, tenant portal — they all need "which units belong to this meter's scope." One view is the answer.

**View shape (sketch, see M0 for the actual SQL):**

```text
meter_scope_resolved(meter_id, unit_id)
  -- For submeters: one row per linked unit (usually exactly one).
  -- For condo-level masters: one row per unit in the condo.
  -- For group-level masters: one row per unit in the group (see [4.11](#411-nested-groups-hierarchy) for how groups nest).
  -- For custom-set masters: one row per linked unit.
```

**Conventions:**

- Hand-written migration in `packages/db/migrations/` (Drizzle's generator won't produce a view).
- Treat the view as part of the schema — any change to `linked_meters` semantics requires updating the view migration in the same PR.
- Always join against `meter_scope_resolved` for consumption math; never re-derive scope ad-hoc in app code.

## 4.6 QR-Detection Library: jsQR

**Decision:** Use [`jsQR`](https://github.com/cozmo/jsQR) as the only QR scanner. No native `BarcodeDetector` path, no `@zxing/browser`.

**Why:**

- **Platform-agnostic by uniformity.** Same JS library on every browser → identical behavior on iOS Safari, Android Chrome, desktop Firefox, etc. No conditional code paths, no edge cases between Chromium's `BarcodeDetector` quirks and a fallback library.
- **We print our own QR codes**, so multi-format support (Code 128, Data Matrix, EAN, …) is dead weight. `@zxing/browser` exists to read existing barcodes from third-party meters; that's not our case.
- **Smallest viable bundle.** `jsQR` is ~40 KB minified — meaningfully lighter than `@zxing/browser` (~50 KB) when we don't need its format zoo.
- **Predictable in tests.** A single library means Playwright runs identically against every browser engine; no `if (BarcodeDetector)` branches to mock around.
- **Simpler ops.** One thing to monitor for vulnerabilities, one thing to upgrade.

**Why not the alternatives:**

- **`BarcodeDetector` (native)** — no iOS Safari support, and subtle behavioral differences across browsers that *do* implement it make tests brittle.
- **`@zxing/browser`** — fine choice, but you pay (in bytes and complexity) for multi-format scanning we'll never use.
- **`html5-qrcode`** — wrapper that bundles its own UI; we want a shadcn-styled scanner that matches the rest of the app.

**Conventions:**

- Implementation in `hidrosync-service/components/qr/QrScanner.tsx`. Camera plumbing (`getUserMedia`, frame capture into `<canvas>`, feed to `jsQR`) is bespoke — about 100 lines.
- Decoded value is always a URL of the form `https://app.hidrosync.com.br/r/<meter_id>`. The component validates the URL shape before navigating; non-HidroSync QRs show a "not a HidroSync meter QR" toast.
- QR **rendering for printing** uses [`qrcode`](https://www.npmjs.com/package/qrcode) on the server (renders to SVG/PNG) — that's a separate library from scanning and stays unchanged.

## 4.7 Onboarding LLM: Claude Sonnet via OpenRouter

**Decision:** Use the current Anthropic Claude Sonnet generation (e.g. `anthropic/claude-sonnet-4`, pinned per env) via OpenRouter for the prompt-driven onboarding parser. Strict JSON output validated server-side with Zod.

**Why:**

- **Onboarding runs once per condo**; cost per call is irrelevant. Pick on quality, not price.
- Sonnet handles **mixed Portuguese + English prompts** (typical for Brazilian property managers) and **ambiguous structural descriptions** ("10 floors, 4 apts each, 1 master per floor + 1 condo intake — except the ground floor which has only commercial units") more reliably than any of the cheaper models.
- A misclassified group at onboarding ruins all subsequent data; Sonnet's accuracy edge is worth paying for here.
- **Same OpenRouter account** as the meter reader → one key, one bill, one rate-limit story.

**Why not the alternatives:**

- **GPT-4o / GPT-5-class** — fine quality, but Anthropic's structured-output adherence has historically been stricter. Keep them as failover.
- **Gemini Flash / cheap models** — too cheap for what's effectively a critical setup step. Reserve those for high-frequency hot paths (meter reading), not low-frequency critical paths (onboarding).
- **Open-source via OpenRouter (Llama, Qwen)** — quality not yet on par for complex Portuguese structured output.

**Conventions:**

- Adapter in `hidrosync-service/modules/ai/OnboardingParser.ts`, mirroring the `MeterReader` interface. `MockOnboardingParser` returns a deterministic structure for dev/tests.
- System prompt + few-shot examples checked in under `hidrosync-service/modules/ai/onboarding-prompts/` so they're versioned alongside code.
- Output is **always** validated against a Zod schema before any DB write; parse failures surface a "couldn't understand the description, please rephrase" message rather than committing nonsense to temp tables.

## 4.8 Email Transport: Resend

**Decision:** Use [Resend](https://resend.com) for all transactional email, with [React Email](https://react.email) for templates.

**Why:**

- **DX is unmatched** for Next.js. React Email lets templates be JSX components that share typography / colors / logo with the app shell — one source of truth for branding.
- **Free tier covers MVP** (3,000 emails/month, 100/day). HidroSync emails are low-volume: invitations, period-open reminders, anomaly notifications. Won't hit a paid tier until many condos.
- **LGPD-compliant DPA available**; EU/regional data-handling controls.
- **Webhooks for delivery state** integrate cleanly into the audit log if/when needed (delivered / bounced / opened).
- **Custom domain setup** (SPF / DKIM / DMARC) is a 5-minute DNS dashboard step — no SES sandbox-removal saga.

**Why not the alternatives:**

- **AWS SES** — cheapest at scale, but the cost difference is irrelevant until volume is high, and the setup tax (domain verification, sandbox removal, IAM) eats time we'd rather spend on product. Migration is straightforward later (transport swap behind the email module).
- **SendGrid** — capable but heavier API and worse template DX. Worth revisiting only if Resend deliverability becomes a problem at scale.
- **Mailgun** — middle ground, no compelling reason to pick it over Resend.

**Conventions:**

- All sends go through `hidrosync-service/modules/notifications/email.ts` — one function `sendEmail({ to, template, props })`.
- Template components live in `hidrosync-service/modules/notifications/templates/*.tsx` (React Email).
- A `MockEmailTransport` writes to the audit log instead of sending in dev/tests.
- Every transactional email ships with a shadcn-styled HTML version + plain-text fallback (React Email gives both).

## 4.9 Image Compression: browser-image-compression + heic2any

**Decision:** Compress photos client-side with [`browser-image-compression`](https://github.com/Donaldcwl/browser-image-compression) before upload. Convert HEIC inputs (iPhone gallery) with [`heic2any`](https://github.com/alexcorvi/heic2any) loaded on demand. Always extract and persist EXIF *before* compression.

**Why:**

- The workflow **requires** EXIF (capture timestamp, GPS, device) on every reading. Compression libraries that re-encode through `<canvas>` typically strip EXIF; `browser-image-compression` has an explicit `preserveExif: true` (via `piexifjs`) that survives the round-trip for JPEGs.
- **Web Worker support is automatic**, so resizing a 12-megapixel iPhone photo doesn't freeze the capture UI.
- **HEIC reality:** iPhone gallery picks deliver HEIC; live camera captures via `<input capture>` deliver JPEG. We only pay the `heic2any` cost (~150 KB libheif WASM, lazy-loaded) when an HEIC is actually selected.
- **EXIF-first pipeline:** parse EXIF from the original file before any conversion. Persist EXIF to the row independently of the photo — even if compression or HEIC conversion strips embedded metadata, the reading row is intact.

**Why not the alternatives:**

- **`compressorjs`** — similar ergonomics but weaker EXIF preservation (some versions drop GPS).
- **Raw `<canvas>` + `OffscreenCanvas`** — you'll re-implement EXIF preservation and HEIC handling yourself.
- **`pica`** — best-in-class resampling quality, lower-level. Overkill for "downscale to 1024 px and ship to GCS / OpenRouter".

**Conventions (M4):**

- One pipeline function `processCapture(file): Promise<{ blob, exif }>` in `hidrosync-service/lib/exif/processCapture.ts`.
- Compression settings: `maxWidthOrHeight: 1024`, `useWebWorker: true`, `initialQuality: 0.85`, `preserveExif: true`.
- The **original full-resolution** photo is *also* uploaded to GCS for audit; only the downscaled copy is sent to OpenRouter.
- HEIC conversion is feature-detected by file extension / MIME, not preemptively imported.

## 4.10 Offline Strategy: @serwist/next + Bespoke IndexedDB Queue

**Decision:** Use [`@serwist/next`](https://serwist.pages.dev/) (modern Workbox-for-Next.js) for shell precaching and runtime cache strategies. Build a **bespoke IndexedDB queue** with foreground sync for offline reading captures.

**Why:**

The offline-first requirement is two unrelated problems, solved separately:

1. **Cache the reading-screen shell** so operators can open `/reading` and `/r/<meter_id>` in a basement with no signal. `@serwist/next` solves this in ~20 lines of config — precache the routes, JS chunks, CSS, and an offline-fallback page.
2. **Queue reading captures (photo blob + EXIF + value) for upload** until the device reconnects. This must be **bespoke** because:
   - Background Sync API is **not supported on iOS Safari**. Workbox's `BackgroundSyncPlugin` silently fails there.
   - Reading payloads are multi-MB photos — they belong in IndexedDB, not the SW request cache.
   - The only reliable cross-platform pattern is a **foreground sync loop** triggered on `visibilitychange`, the `online` event, and a periodic timer while the tab is open.

**Why not the alternatives:**

- **`next-pwa`** — in maintenance mode; `@serwist/next` is the actively-maintained successor.
- **Workbox without a Next.js wrapper** — fights Next's build pipeline; `@serwist/next` handles it.
- **Pure hand-rolled SW** — re-implementing cache-first / network-first strategies from scratch is wasted effort when battle-tested abstractions exist.

**Conventions (M4):**

- `@serwist/next` config precaches: the reading-screen shell, the QR-scanner page, and the offline fallback. Runtime strategy for API routes is `NetworkFirst` with a short timeout, falling back to the offline page.
- IndexedDB store `pending_readings` holds queued captures, keyed by a client-generated UUID.
- A `useOfflineQueue()` hook reads/writes the store and exposes queue state to the UI ("3 readings pending sync").
- Foreground sync hook fires on:
  1. `visibilitychange` (tab returns to foreground)
  2. `online` event (network reconnects)
  3. A 30-second interval while the tab is open
- On supported browsers, **also** register Background Sync as belt-and-suspenders.
- The capture API endpoint is **idempotent** (server upserts on `(meter_id, period_id, client_id)`) so re-sends from a flaky queue don't double-count.

## 4.11 Nested `groups` hierarchy

**Decision:** Represent condo subdivisions as a **tree of `groups`** with optional `parent_group_id` (self-FK to `groups.id`, `ON DELETE CASCADE`). `NULL` means a **root** group. Cap **depth at four levels** counting the root, enforce that limit in the **onboarding module** when building `temp_groups` and permanent `groups` rows (not only with implicit DB constraints). Enforce **name uniqueness** with Postgres **partial unique indexes**: among roots in a condo, `(condo_id, name)` is unique where `parent_group_id IS NULL`; among siblings under the same parent, `(condo_id, parent_group_id, name)` is unique where `parent_group_id IS NOT NULL`. Apply the **same rules** to **`temp_groups`**, scoped by `session_id` (see migration `0002_*`). **`units.group_id`** remains a **single** FK to one group — typically the **leaf** group the unit belongs to.

**Why:**

- **Depth and naming invariants** belong in onboarding because that is where bulk structure is created; failing fast there avoids impossible trees and duplicate labels before commit.
- **Partial unique indexes** express “unique among roots” vs “unique among siblings” without nullable-column uniqueness footguns.
- **Parity between `temp_groups` and `groups`** keeps preview/commit idempotent and avoids session-local collisions.
- **One `group_id` per unit** keeps unit placement simple for billing and for `meter_scope_resolved` when masters target a group.

**Conventions:**

- When changing group shape or validation, update Drizzle schema, migrations (indexes), onboarding builders, and this section in the same PR.
